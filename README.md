# Order API

A Rust REST API for managing orders, built following the **Model-First Development** pipeline where code — not specs — is the source of truth.

**Stack:** Rust, Actix-web, Diesel (Postgres), utoipa (OpenAPI), just

## Pipeline

Everything flows in one direction. The pipeline order is a hard invariant.

```
SQL Migration → Rust Model → Handler + utoipa → Route Registration
                                                        │
                                                 GENERATION BOUNDARY
                                                        │
                                                  openapi.json → TS client → Frontend
```

Everything above the generation boundary is hand-written, authoritative code.
Everything below is derived and can be regenerated with `just gen`.

## Domain

**Orders** with a status state machine and **line items** with computed totals.

```
Order
├── id, status, currency, total_amount
├── status: Draft → Confirmed → Shipped → Delivered
│                 ↘ Cancelled ↙
└── OrderLineItem (1:N)
    ├── product_sku, quantity, unit_price
    └── line_total (generated column: quantity × unit_price)
```

### Key Rules

- `NUMERIC(19,4)` precision is preserved end-to-end (DB → Rust `BigDecimal` → JSON string)
- Line items can only be added/removed while order is in `Draft`
- `confirmed_at` is set automatically on `Draft → Confirmed` transition
- `total_amount` is recomputed from line items on every add/delete
- Cancellation is only valid from `Draft` or `Confirmed`

## Prerequisites

- [Rust](https://rustup.rs/) (stable)
- [just](https://github.com/casey/just) — `cargo install just`
- [Diesel CLI](https://diesel.rs/guides/getting-started) — `cargo install diesel_cli --no-default-features --features postgres`
- Docker (for Postgres)

## Setup

```bash
# 1. Clone and enter the project
cd model-first-order

# 2. Create .env from the example
cp .env.example .env

# 3. Start Postgres
just db

# 4. Run migrations
just migrate

# 5. Start the server
just dev
```

The API starts at `http://localhost:8080`. Swagger UI is at `http://localhost:8080/swagger-ui/`.

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/health` | Health check |
| `POST` | `/api/orders` | Create a draft order |
| `GET` | `/api/orders` | List all orders |
| `GET` | `/api/orders/{id}` | Get order with line items |
| `PATCH` | `/api/orders/{id}/status` | Transition order status |
| `POST` | `/api/orders/{id}/items` | Add a line item |
| `DELETE` | `/api/orders/{order_id}/items/{item_id}` | Remove a line item |

## Just Commands

```bash
just dev             # Run the API server
just db              # Start Postgres (Docker Compose)
just db-stop         # Stop Postgres
just migrate         # Run Diesel migrations
just migrate-revert  # Revert last migration
just db-reset        # Reset database (revert all + re-run)
just gen             # Export openapi.json (generation boundary)
just test            # Run tests
just lint            # Run clippy
just fmt             # Format code
just pre-commit      # fmt check + clippy + test
```

## Pipeline Violations

Do not do these — they break the single-direction flow:

| Violation | Why |
|-----------|-----|
| Edit `schema.rs` by hand | Generated by Diesel — overwritten on next `diesel migration run` |
| Edit `openapi.json` by hand | Generated from utoipa — overwritten on next `just gen` |
| Write handler before migration + model | Handler references types that won't compile |
| Skip `just gen` after model/handler changes | TS client becomes stale |
| Skip `just pre-commit` before pushing | Untested code reaches the branch |
