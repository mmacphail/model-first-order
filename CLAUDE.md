## Build & Development Commands

```bash
just dev              # Start API with auto-reload (cargo-watch)
just db               # Start Postgres via Docker Compose
just migrate          # Run Diesel migrations
just db-reset         # Revert all migrations + re-run
just infra-up         # Start full infra (Postgres, Kafka, Schema Registry, Debezium, AKHQ)
just gen              # Export openapi.json from utoipa annotations
just smoke            # Quick manual smoke test (create order, add item, confirm)
```

## Architecture

**Model-First pipeline** — code flows in one direction, never backwards:

```
SQL Migration → Rust Model → Handler + utoipa → Route Registration
                                                      │
                                               GENERATION BOUNDARY
                                                      │
                                                openapi.json → TS client
```

Everything above the boundary is hand-written and authoritative. Everything below is derived via `just gen`.

**Stack:** Rust, Actix-web, Diesel (Postgres, synchronous via `web::block`), utoipa (OpenAPI), just

### Key Modules

- `src/models/` — Diesel structs (`Queryable`, `Insertable`) and domain logic. `order_status.rs` contains the state machine (`can_transition_to`) and outbox event type constants.
- `src/handlers/orders.rs` — All order/line-item endpoints. Each mutating handler runs inside a `conn.transaction()` that writes business data + outbox event atomically.
- `src/models/outbox.rs` — `insert_outbox_event()` loads the full `OrderWithItems` aggregate, serializes it to JSON, and inserts into `commerce_order_outbox`. Must be called inside the caller's transaction.
- `src/schema.rs` — Auto-generated by Diesel CLI. **Never hand-edit.**
- `src/serializers.rs` — `BigDecimal` ↔ String serde for NUMERIC(19,4) precision preservation.
- `src/errors.rs` — `ApiError` enum implementing Actix `ResponseError`. Diesel errors auto-convert.

### Domain Rules

- Order status state machine: Draft → Confirmed → Shipped → Delivered, with Cancelled allowed from Draft or Confirmed only.
- Line items can only be added/removed while order is `Draft`.
- `total_amount` is recomputed from `line_total` (DB generated column: quantity × unit_price) on every add/delete.
- `confirmed_at` is set on Draft → Confirmed transition.
- Every mutating operation writes a transactional outbox event (CDC via Debezium → Kafka).

### Testing

Integration tests (`tests/order_lifecycle.rs`) use **testcontainers** to spin up a disposable Postgres per test. Docker must be running. Tests use Actix's `test::init_service` with the real router — no mocking.

E2E CDC tests (`tests/e2e_test.rs`, run via `just test-e2e`) start the full Docker Compose stack.

### Infrastructure

`infra/docker-compose.yml` defines: Postgres (debezium/postgres:16 with `wal_level=logical`), Kafka (KRaft mode), Confluent Schema Registry, Debezium Connect, AKHQ (Kafka UI at :8090).

### Pipeline Violations to Avoid

- Don't edit `src/schema.rs` — it's overwritten by `diesel migration run`
- Don't edit `openapi.json` — it's overwritten by `just gen`
- Don't write handlers before migration + model (won't compile)
- Run `just gen` after any model/handler change
