# Multi-Agent Development Guide

This guide covers how multiple AI agents (Claude Code sessions) can work safely and efficiently in parallel on this codebase.

## Core Principle: Worktrees for Isolation

Every agent session that modifies code MUST work in its own git worktree. Claude Code creates these automatically in `.claude/worktrees/<name>/`. Each worktree is a full checkout with its own working tree and index, sharing the same git object database.

**Why this matters:**
- Each worktree has its own `target/` directory — parallel `cargo build` and `cargo test` do not conflict.
- Each worktree has its own git branch — commits do not collide.
- Testcontainers allocates random ports per test — parallel test suites do not conflict.
- The shared `.env` is not committed (`.gitignore`), so each worktree can have its own database URL if needed.

## Module Boundaries and Ownership

When multiple agents work in parallel, they should operate on **non-overlapping modules** to avoid merge conflicts. The codebase has clear boundaries:

### High-Conflict Zones (coordinate before touching)

| File / Module | Why it's shared | Coordination needed |
|---|---|---|
| `src/routes.rs` | Every new endpoint must be registered here | Add routes in a single PR; do not have 2 agents both adding routes |
| `src/lib.rs` | New top-level modules are declared here | Same — one agent at a time |
| `src/handlers/mod.rs` | New handler modules declared here | Same |
| `src/models/mod.rs` | New model modules declared here | Same |
| `src/schema.rs` | Auto-generated by Diesel — never hand-edit | Only one agent should run `diesel migration run` at a time |
| `openapi.json` | Auto-generated by `just gen` — never hand-edit | Regenerate as last step before PR |
| `Cargo.toml` | Adding dependencies | Coordinate to avoid version conflicts |

### Safe for Parallel Work (one agent per file)

| Module | What it contains | Independent? |
|---|---|---|
| `src/handlers/orders.rs` | Order endpoints | Yes, if not adding new routes |
| `src/models/order.rs` | Order struct/queries | Yes |
| `src/models/order_status.rs` | State machine | Yes |
| `src/models/order_line_item.rs` | Line item struct | Yes |
| `src/models/outbox.rs` | Outbox event logic | Yes |
| `src/errors.rs` | Error types | Yes, add new variants at end |
| `src/serializers.rs` | BigDecimal serde | Yes |
| `tests/order_lifecycle.rs` | Integration tests | Yes (testcontainers isolates DB) |
| `tests/e2e_test.rs` | E2E tests | No — uses shared Docker Compose stack |
| New `src/handlers/<name>.rs` | New feature handler | Yes — fully independent until route registration |
| New `src/models/<name>.rs` | New feature model | Yes — fully independent until mod.rs registration |

### Migrations Deserve Special Care

Diesel migrations are **ordered by timestamp prefix**. Two agents creating migrations simultaneously will:
1. Both generate unique timestamps (safe).
2. Both modify `src/schema.rs` when run (conflict-prone).

**Rule:** Only one agent should create and run a migration at a time. If two features need migrations, sequence them: Agent A creates migration and merges, then Agent B creates theirs.

## Branch Naming Convention

Agent-created branches should follow this pattern:

```
<type>/<short-description>
```

Examples:
- `feat/add-product-aggregate`
- `fix/outbox-serialization-bug`
- `refactor/extract-pagination`
- `test/status-transition-coverage`

The `<type>` matches the Conventional Commits type that the PR will use. This makes it clear what each agent is working on when listing branches.

## Workflow: Adding a New Feature (Multi-Agent)

A large feature (e.g., "add Product aggregate with CRUD + outbox") can be decomposed:

### Phase 1 — Sequential (single agent)
1. Write the SQL migration (`migrations/<timestamp>_create_products/`)
2. Run `diesel migration run` to update `src/schema.rs`
3. Register new modules in `src/models/mod.rs`, `src/handlers/mod.rs`, `src/lib.rs`
4. Register routes in `src/routes.rs`
5. Commit and push this "skeleton" branch

### Phase 2 — Parallel (multiple agents, each in a worktree)
- **Agent A:** Implement `src/models/product.rs` (struct, queries, validation)
- **Agent B:** Implement `src/handlers/products.rs` (endpoint logic, utoipa annotations)
- **Agent C:** Write tests in `tests/product_lifecycle.rs`
- **Agent D:** Update documentation

### Phase 3 — Sequential (single agent)
1. Merge all agent branches (resolve any conflicts in shared files)
2. Run `just quality` to verify everything
3. Run `just gen` to regenerate `openapi.json`
4. Create the final PR

## Quality Gate Safety

`just quality` runs: `just gen` then `cargo fmt --check` then `cargo clippy` then `cargo test`.

**Is it safe to run in parallel across worktrees?** Yes, because:
- Each worktree has its own `target/` directory — no build artifact conflicts.
- `cargo test` uses testcontainers, which spins up a disposable Postgres per test on a random port.
- `just gen` writes `openapi.json` local to the worktree.
- `cargo fmt --check` and `cargo clippy` are read-only.

**Exception:** `just test-e2e` is NOT safe to run in parallel. It uses a shared Docker Compose stack with fixed ports (5432, 9092, 8081, 8083, 8090). Only one agent should run E2E tests at a time.

## Commit Conventions for Agents

All agents MUST follow Conventional Commits (see `.claude/rules/commit-conventions.md`). Additional conventions for multi-agent work:

- Each agent's commits should be atomic and self-contained.
- If an agent's work depends on another agent's branch, rebase onto it before creating a PR.
- Commit messages should include the scope to make authorship clear: `feat(products): add CRUD handler`.
- Never force-push a branch that another agent might be tracking.

## Shared Scratchpad

Agents can leave notes for each other in `.claude/scratchpad/`. This directory is gitignored and intended for ephemeral coordination:

- `STATUS.md` — Each agent can append their current status (what they are working on, blockers).
- `DECISIONS.md` — Record architectural decisions made during the session.
- `TODO.md` — Tasks discovered during work that should be picked up later.

These files are not committed. They exist only for the duration of a parallel work session.

## Pre-Merge Checklist

Before any agent's branch is merged to main:

1. `just quality` passes (format + lint + test)
2. No uncommitted changes to `src/schema.rs` or `openapi.json`
3. Branch is rebased onto latest `main`
4. All new public types have utoipa `ToSchema` derives
5. All new endpoints have utoipa `#[utoipa::path]` annotations
6. All mutating endpoints write outbox events inside their transaction
7. Commit messages follow Conventional Commits
